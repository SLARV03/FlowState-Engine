# FlowState-Engine — System Prompt & Agent Definitions

> **Role:** High-Performance Multi-Agent Orchestration Engine  
> **Codename:** FlowState-Engine  
> **Version:** 0.1.0-alpha  
> **Last Updated:** 2026-05-19

---

## 1. Mission Statement

FlowState-Engine is a **stateful, graph-based autonomous software engineering workspace** where specialized AI agents collaborate to write source files, execute tests inside an ephemeral Docker sandbox, and iteratively fix bugs until a user-defined prompt passes QA — all orchestrated through a finite state machine with real-time UI feedback.

---

## 2. System Architecture Overview

The execution framework operates as a **Stateful Finite State Machine (FSM)**. Unlike linear LLM chains, the global system state is passed **mutable-by-reference** through a directed graph (implemented via LangGraph). Each node in the graph represents an agent invocation, and edges represent conditional routing decisions based on the current state.

### 2.1 High-Level Data Flow

```
┌──────────────┐     ┌──────────────┐     ┌──────────────┐     ┌──────────────┐
│   User Prompt│────▶│  PM Agent    │────▶│  SWE Agent   │────▶│  QA Agent    │
│   (START)    │     │  (Spec Gen)  │     │  (Code Gen)  │     │  (Test Gen)  │
└──────────────┘     └──────────────┘     └──────────────┘     └──────┬───────┘
                                                                      │
                                                ┌─────────────────────┤
                                                │                     │
                                          ┌─────▼─────┐        ┌─────▼─────┐
                                          │  Sandbox   │        │  Docker   │
                                          │  Runner    │◀───────│  Runtime  │
                                          └─────┬──────┘        └───────────┘
                                                │
                              ┌─────────────────┼─────────────────┐
                              │                 │                 │
                        ┌─────▼─────┐     ┌─────▼─────┐    ┌─────▼─────┐
                        │  PASSED   │     │  FAILED   │    │  MAX ITER │
                        │  → Deploy │     │  → SWE    │    │  → Human  │
                        │  → END    │     │  (retry)  │    │  Breakout │
                        └───────────┘     └───────────┘    └───────────┘
```

## 3. Global State Schema

```python
from typing import TypedDict, List, Dict, Any

class SquadState(TypedDict):
    user_requirement: str          # Original prompt from user
    technical_spec: str            # Markdown spec generated by PM
    file_system: Dict[str, str]    # Virtual workspace: {"path/to/file.py": "code_content"}
    test_suite: Dict[str, str]     # Unit tests: {"tests/test_file.py": "test_content"}
    latest_test_logs: str          # Terminal stdout/stderr from the Docker execution
    test_status: str               # "PENDING" | "PASSED" | "FAILED"
    iteration_count: int           # Safety breaker to prevent infinite loop recursion
    current_agent: str             # Tracking active node for UI WebSockets
    chat_history: List[Dict[str, str]]  # Internal agent communication logs
```

### 3.1 State Field Semantics

| Field               | Initial Value | Mutated By     | Description                                        |
| ------------------- | ------------- | -------------- | -------------------------------------------------- |
| `user_requirement`  | User input    | Never          | Immutable seed prompt from the user                |
| `technical_spec`    | `""`          | PM Agent       | Comprehensive markdown specification               |
| `file_system`       | `{}`          | SWE Agent      | File paths → source code content                   |
| `test_suite`        | `{}`          | QA Agent       | Test file paths → test code content                |
| `latest_test_logs`  | `""`          | Sandbox Runner | Raw stdout/stderr from Docker execution            |
| `test_status`       | `"PENDING"`   | Sandbox Runner | Enum: `PENDING`, `PASSED`, `FAILED`                |
| `iteration_count`   | `0`           | Sandbox Runner | Incremented on each test execution cycle           |
| `current_agent`     | `"pm"`        | Router         | Active node for WebSocket UI updates               |
| `chat_history`      | `[]`          | All Agents     | Append-only inter-agent communication log          |

## 4. Agent Archetypes & System Prompts

Each agent is an **independent LLM instantiation** with a role-specific system prompt and a restricted set of programmatic tools.

---

### 4.1 Agent A — The Product Manager (PM)

**Node ID:** `pm_agent` · **Temperature:** `0.4` · **Max Tokens:** `4096`

#### System Prompt

```
You are an elite Technical Product Manager working inside the FlowState-Engine
multi-agent orchestration system. Your job is to take raw, ambiguous user
requirements and translate them into an unassailable, implementation-ready
markdown technical specification.

Your specification MUST include:
1. A clear project title and one-paragraph summary
2. Core features broken into discrete, implementable modules
3. Input/output contracts for each module
4. Edge cases and error handling expectations
5. Architectural expectations (file structure, language, patterns)
6. Explicit acceptance criteria that the QA Engineer can convert into tests

CONSTRAINTS:
- Do NOT write any code. Your output is pure specification.
- Do NOT assume language/framework unless stated. Default to Python.
- Be exhaustive. Every unresolved ambiguity cascades into downstream bugs.
```

#### Permitted Tools

| Tool            | Signature                        | Effect                                              |
| --------------- | -------------------------------- | --------------------------------------------------- |
| `submit_spec`   | `(markdown_string: str) -> None` | Writes to `state.technical_spec`, advances to SWE   |

#### Required Output Sections

`## Project Overview` · `## Module Breakdown` · `## API / Interface Contracts` · `## Edge Cases & Error Handling` · `## File Structure` · `## Acceptance Criteria`

---

### 4.2 Agent B — The Software Engineer (SWE)

**Node ID:** `swe_agent` · **Temperature:** `0.2` · **Max Tokens:** `8192`

#### System Prompt

```
You are a Senior Full-Stack Software Engineer working inside the FlowState-Engine
multi-agent orchestration system. Read the technical specification and write
clean, modular, documented code into the virtual file system.

FIRST ITERATION: Read technical_spec, design file/module structure, write all
source files via write_file(), ensure docstrings and type hints, call submit_to_qa().

BUG-FIX ITERATION (latest_test_logs non-empty): Treat failures as HIGH-PRIORITY
bugs. Analyze stack traces, identify root causes, use patch_file() for targeted
fixes or write_file() for rewrites. Do NOT mock functionality. Call submit_to_qa().

CONSTRAINTS:
- Do NOT write test files (QA's responsibility).
- Do NOT modify test files under any circumstances.
- Write production-quality code. No placeholder TODO stubs.
```

#### Permitted Tools

| Tool           | Signature                                            | Effect                                       |
| -------------- | ---------------------------------------------------- | -------------------------------------------- |
| `write_file`   | `(path: str, content: str) -> None`                  | Creates/overwrites file in `state.file_system` |
| `patch_file`   | `(path: str, search: str, replace: str) -> None`     | Targeted string replacement in existing file |
| `submit_to_qa` | `() -> None`                                         | Concludes coding, advances graph to QA node  |

### 4.3 Agent C — The QA Engineer (QA)

**Node ID:** `qa_agent` · **Temperature:** `0.1` · **Max Tokens:** `6144`

#### System Prompt

```
You are a rigorous Automation QA Engineer working inside the FlowState-Engine
multi-agent orchestration system. Review the SWE's code and write comprehensive
automated tests.

RESPONSIBILITIES:
1. Read technical_spec for acceptance criteria.
2. Read file_system to understand implementation.
3. Write test files via write_test_file() covering: happy path, edge cases,
   input validation, error handling, and performance constraints.
4. Call run_test_sandbox() to execute tests in Docker sandbox.

FRAMEWORK: Python→pytest, JS/TS→Jest/Vitest, Go→testing package.

CONSTRAINTS:
- Do NOT modify source code files. Only write test files.
- Do NOT mock dependencies unless absolutely necessary.
- Use descriptive names: test_<feature>_<scenario>.
- Include positive and negative test cases.
- On re-iterations, ADD tests but do NOT remove existing ones.
```

#### Permitted Tools

| Tool               | Signature                           | Effect                                                         |
| ------------------ | ----------------------------------- | -------------------------------------------------------------- |
| `write_test_file`  | `(path: str, content: str) -> None` | Writes test script to `state.test_suite`                       |
| `run_test_sandbox` | `() -> SandboxResult`               | Docker execution; mutates logs, status, increments iteration   |

## 5. Execution Loop & Conditional Routing

### 5.1 Routing Logic

```python
def route_after_qa(state: SquadState) -> str:
    if state["test_status"] == "PASSED":
        return "deployment_node"           # → END
    elif state["test_status"] == "FAILED" and state["iteration_count"] < 5:
        return "swe_agent"                 # → Bug-fix loop
    else:
        return "human_intervention_node"   # → Safety breaker
```

### 5.2 Graph Definition

```python
from langgraph.graph import StateGraph, END

graph = StateGraph(SquadState)
graph.add_node("pm_agent", pm_agent_fn)
graph.add_node("swe_agent", swe_agent_fn)
graph.add_node("qa_agent", qa_agent_fn)
graph.add_node("deployment_node", deployment_fn)
graph.add_node("human_intervention_node", human_intervention_fn)

graph.set_entry_point("pm_agent")
graph.add_edge("pm_agent", "swe_agent")
graph.add_edge("swe_agent", "qa_agent")
graph.add_conditional_edges("qa_agent", route_after_qa, {
    "deployment_node": "deployment_node",
    "swe_agent": "swe_agent",
    "human_intervention_node": "human_intervention_node",
})
graph.add_edge("deployment_node", END)
graph.add_edge("human_intervention_node", END)

app = graph.compile()
```

### 5.3 Visual Pipeline

```
[START] → [PM] → [SWE] → [QA] → [Sandbox]
                    ▲                 │
                    │    FAILED       │
                    │    (iter<5)     │
                    └─────────────────┤
                                      ├── PASSED → [DEPLOY] → [END]
                                      └── MAX_ITER → [HUMAN] → [END]
```

## 6. Sandbox Layer — Docker Execution Security

### 6.1 Execution Workflow

1. **Directory Prep:** Create `/tmp/flowstate_session_{uuid}/`, dump `file_system` + `test_suite`.
2. **Container Spin-Up:** Select base image by language, mount as read-only volume.
3. **Execute:**
   ```bash
   docker run --rm --network none --memory 256m --cpus 0.5 \
     --read-only --tmpfs /tmp:size=64m --cap-drop ALL --pids-limit 64 \
     -v /tmp/flowstate_session_{uuid}:/app:ro -w /app \
     python:3.11-alpine \
     sh -c "pip install -r requirements.txt 2>/dev/null; pytest -v --tb=short 2>&1"
   ```
4. **Capture:** stdout+stderr, enforce 30s timeout, parse exit code → `PASSED`/`FAILED`.
5. **Cleanup:** Remove temp directory (container auto-removed via `--rm`).

### 6.2 Security Constraints

| Constraint        | Value              | Rationale                              |
| ----------------- | ------------------ | -------------------------------------- |
| Network           | `--network none`   | Prevent data exfiltration              |
| Memory            | `256 MB`           | Prevent OOM attacks                    |
| CPU               | `0.5 cores`        | Prevent resource exhaustion            |
| Filesystem        | Read-only + tmpfs  | Prevent persistent mutations           |
| Timeout           | `30 seconds`       | Prevent infinite loops                 |
| Capabilities      | `--cap-drop ALL`   | Minimize kernel attack surface         |
| PID Limit         | `64`               | Prevent fork bombs                     |

## 10. Error Handling & Safety

### 10.1 Iteration Safety Breaker

Hard limit of **5 iterations** (configurable). On breach: route to Human Intervention Node → WebSocket `error` event → present accumulated logs → allow user to modify prompt, edit files, adjust spec, or abort.

### 10.2 LLM Failure Recovery

| Failure               | Recovery                                          |
| ---------------------- | ------------------------------------------------- |
| Rate limit (429)       | Exponential backoff with jitter (max 3 retries)   |
| Token limit exceeded   | Truncate/summarize `chat_history`                 |
| Malformed tool call    | Re-prompt agent with error + tool schema          |
| Timeout                | Kill container, log error, increment iteration    |

### 10.3 Cost Tracking

Track cumulative token usage per session (input/output tokens per agent, estimated USD cost) — displayed in UI sidebar as running total.
